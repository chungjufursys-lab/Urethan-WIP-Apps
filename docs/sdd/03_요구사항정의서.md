# 03. 요구사항정의서

## 기능 요구사항
- `FR-001` 시스템은 `정규화 부품명 + 색상`을 모든 조회와 관리의 기본 KEY로 사용해야 한다.
- `FR-002` 시스템은 업로드 원본의 `제품코드`, `제품색상`, `부품코드`, `부품색상`, `부품명`, `최초투입일자`, `포장일자`를 모두 보존해야 한다.
- `FR-003` 시스템은 업로드 시 `부품명`에서 `[C20~]`과 같은 문자열을 제거한 `정규화 부품명`을 생성해야 한다.
- `FR-004` 시스템은 제품코드와 부품코드의 N:1 관계를 매핑 테이블로 관리해야 한다.
- `FR-005` 시스템은 위험도 계산 기준 날짜로 `최초투입일자`를 사용해야 한다.
- `FR-006` 시스템은 화면 노출 기준 날짜로 `포장일자`를 사용해야 한다.
- `FR-007` 시스템은 생산계획 저장 시 제품 정보, 부품 정보, 정규화 부품명을 독립 컬럼으로 관리해야 한다.
- `FR-008` 시스템은 캘린더 및 조회 화면에서 기준 부품 외에 보조 정보로 `제품코드`를 노출할 수 있어야 한다.
- `FR-009` 시스템은 현재 구조와 목표 구조의 매핑 차이를 문서로 추적 가능해야 한다.

## 비기능 요구사항
- `NFR-001` 데이터 구조 문서는 현재 상태와 목표 상태를 구분해 기록해야 한다.
- `NFR-002` 미확정 항목은 추정하지 않고 `미정` 또는 `사용자 확인 필요`로 남겨야 한다.
- `NFR-003` 용어는 제품 정보, 부품 정보, 정규화 부품명을 혼동 없이 구분해야 한다.

## 현재 확인된 구조 충돌
- `CR-001` 현재 `production_plan`은 `product_code`를 저장하지만 실제 서비스 로직은 `urethane_item_code`를 `item_code`로 바꿔 사용한다.
- `CR-002` 현재 `color` 컬럼은 실질적으로 `부품색상` 역할을 한다.
- `CR-003` 현재 조회 키는 사실상 `부품코드 + 부품색상`인데, 동일 부품에 대한 명칭 정규화가 없다.
- `CR-004` 현재 제품코드와 부품코드의 연결 관계를 유지하는 전용 매핑 구조가 없다.
- `CR-005` 현재 `business_key`는 사용자 정의 관리 KEY가 아니라 내부 업로드/저장 식별 용도로 혼용된다.

## 미확정 요구사항
- 현재 단계에서 필수 미확정 요구사항 없음
