# 04. 데이터정의서

## 입력 원본
- 파일명: `가공실적등록.xls`
- 성격: 부품단위 계획 파일
- 확인된 입력 컬럼
- `부품이동카드번호`
- `최초투입일자`
- `부품코드`
- `부품색상`
- `부품명`

- `생산수량`

- `제품코드`
- `제품색상`
- `제품명`

## 확정 설계 방향
- 별도 BOM 테이블은 두지 않는다.
- 현재 문제는 제품코드와 부품명 기준 관리 구조를 별도로 정리하지 않은 점이다.
- 기존 부품코드의 역할은 정규화 부품명으로 대체한다.
- 제품코드와 정규화 부품명의 연결을 위한 매핑 테이블이 필요하다.
- 관리 기준 KEY는 `정규화 부품명 + 색상`이다.
- 모든 노출 테이블은 `정규화 부품명 + 색상` 기준으로 관리한다.
- 캘린더 노출에는 `제품코드`를 보조 정보로 추가한다.
- 업체 매핑은 내부적으로 부품코드 또는 부품명 어느 쪽을 사용해도 되지만, 노출 기준은 `정규화 부품명 + 색상`이다.
- `부품명`은 파싱 시 대괄호로 둘러싸인 문자열을 모두 제거한 뒤 저장/조회에 사용해야 한다.
- 동일 부품에 여러 일정이 있으면 화면에서는 합산 1행으로 보여준다.

## 현재 DB 구조 진단

### production_plan 현재 상태
- `product_code`: 제품코드 저장
- `product_name`: 제품명 저장
- `urethane_item_code`: 현재는 실제 부품코드 저장
- `color`: 실제 부품색상 저장
- `plan_date`: 현재 `최초투입일자`
- `due_date`: 현재 `포장일자`
- `required_qty`: 현재 `계획량 - 생산수량` 기반
- 문제점: 정규화 부품명 컬럼이 없고, 제품코드와 정규화 부품명의 연결을 유지하는 구조가 없다.

### services 계층 현재 사용 방식
- `urethane_item_code`를 `item_code`로 변경해 사용한다.
- 이후 부족 계산, 업체 매핑, 캘린더 조회, 재공 비교가 모두 `item_code + color`로 수행된다.
- 따라서 실제 조회/집계 기준은 `부품코드 + 부품색상`이다.
- 목표 구조에서는 이를 `정규화 부품명 + 색상` 기준으로 전환한다.

### inventory 현재 상태
- `item_code`, `color`만 저장한다.
- 현재 문서상 의미는 품목처럼 보이지만, 실제 계획 테이블과 연결될 때는 부품 기준으로 해석된다.

## 목표 데이터 구조 초안

### 목표 공통 원칙
- 조회 KEY: `part_name_normalized + part_color`
- 보조 식별자: `product_code`
- 날짜 의미 분리
- `first_input_date`: 위험도 계산 기준
- `package_date`: 화면 노출 기준

- `product_color`
- `product_name`
- `part_color`
- `part_name_raw`
- `part_name_normalized`
- `first_input_date`

- `produced_qty`
- `required_qty`
- `plan_status`
- `source_type`

### 목표 code_mapping 초안
- `id`
- `product_code`
- `product_color`
- `product_name`
- `part_color`
- `part_name_normalized`
- `updated_at`

### 목표 item_vendor_map 초안
- `id`
- `part_name_normalized`
- `part_color`
- `vendor_name`
- `updated_at`
- 노출 기준 컬럼: `part_name_normalized`, `part_color`

### 목표 inventory 초안
- `id`
- `base_date`
- `part_name_normalized`
- `part_color`
- `qty`
- `location_code`
- `status`
- `remark`
- `updated_by`
- `updated_at`

### 목표 조회/집계 관점
- 기본 조회 화면은 `part_name_normalized + part_color` 기준
- 상세에서 연관 `product_code` 목록을 함께 제공
- 위험도 계산은 `first_input_date` 기준
- 화면 대표 날짜는 `package_date` 기준
- 캘린더에는 `product_code`를 함께 노출
- 동일 `part_name_normalized + part_color`에 다중 일정이 있으면 화면에서는 합산 1행으로 제공

### 부품명 정규화 규칙 초안
- 원본 `부품명`은 `part_name_raw`로 보존한다.
- 조회/집계용 `part_name_normalized`를 생성한다.
- `part_name_normalized` 생성 시 대괄호로 둘러싸인 문자열을 모두 제거한다.
- 제거 후 앞뒤 공백은 trim 처리한다.
