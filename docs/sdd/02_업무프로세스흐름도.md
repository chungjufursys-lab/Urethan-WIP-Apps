# 02. 업무프로세스흐름도

## 시작 조건
- 관리자가 `가공실적등록.xls`를 업로드하거나 기본 파일을 다시 불러온다.

## 현재 상태 업무 흐름
1. 업로드 파일에서 제품 정보, 부품 정보, 날짜, 수량을 읽는다.
2. 현재 코드에서는 `부품코드 -> urethane_item_code`, `부품색상 -> color`로 저장한다.
3. 이후 부족 계산, 위험도 계산, 업체 매핑 조회는 저장된 `item_code + color` 기준으로 수행된다.

## 현재 상태 문제 흐름
1. 제품코드와 부품코드의 N:1 관계를 별도로 유지하지 않는다.
2. 동일 부품이어도 부품명 표기 차이를 정규화하지 않는다.
3. 조회 기준과 화면 노출 정보가 분리되어 있지 않다.

## 목표 업무 흐름
1. 업로드 시 제품 정보와 부품 정보를 모두 독립 컬럼으로 저장한다.
2. 업로드 시 `부품명`에서 `[C20~]` 같은 문자열을 제거해 `정규화 부품명`을 만든다.
3. 생산계획 관리와 조회의 기본 기준은 `정규화 부품명 + 색상`으로 통일한다.
4. 제품코드와 부품코드의 연결 관계는 별도 매핑 테이블에 유지한다.
5. 위험도는 `최초투입일자` 기준으로 계산한다.
6. 화면 표시 기준 날짜는 `포장일자`를 사용한다.
7. 캘린더에는 보조 정보로 `제품코드`를 함께 표시한다.

## 예외 흐름
- 업로드 파일에 필수 컬럼이 없으면 업로드를 중단해야 하는지: 사용자 확인 필요
- 동일 정규화 부품명인데 부품코드가 여러 개인 경우 처리 규칙: 사용자 확인 필요
- 동일 `정규화 부품명 + 색상`에 서로 다른 `포장일자`가 여러 건 존재할 때 표시 규칙: 미정

## 종료 조건
- 관리 기준 KEY, 매핑 구조, 날짜 기준이 서로 충돌하지 않는 구조가 문서로 확정된다.
